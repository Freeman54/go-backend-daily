# 用 optional field 建模 PATCH 三态字段

后端接口做局部更新时，最容易混淆的不是字段类型，而是字段语义：调用方到底是“没传这个字段”、“明确传了 `null` 要清空”，还是“传了一个新值要覆盖”？如果只用普通结构体接 JSON，这三种状态很容易被压扁成两种，导致接口行为不稳定。

## 设计要点

- 为每个可 PATCH 字段保留 `Present / Null / Value` 三态信息
- 缺失字段不参与更新，`null` 表示显式清空，具体值表示覆盖
- 把三态解析放在统一类型里，避免业务层到处写重复判断

## 示例说明

示例中的 `Field[T]` 通过自定义 `UnmarshalJSON` 区分字段是否出现、是否为 `null`。`Apply` 展示了如何把这种三态语义映射到用户资料更新逻辑。测试覆盖了缺失、`null`、具体值三种输入状态，以及局部更新和显式清空行为。

## 运行方式

```bash
go test ./daily/2026/07/07/optional_field_patch
```
