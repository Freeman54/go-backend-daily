# 用 backpressure controller 控制消息消费节奏

消息队列消费者最怕两种失控状态：一是下游处理速度跟不上，`in-flight` 持续堆高；二是业务峰值过大，消费 lag 越积越多。如果只靠固定并发数而没有暂停 / 恢复机制，消费者要么把自己压垮，要么让重试和超时滚雪球。

## 设计要点

- 同时观察 `lag` 与 `in-flight` 两个指标，而不是只盯一个数字
- 使用 pause / resume 双阈值形成迟滞，避免状态来回抖动
- 把“是否继续拉消息”收敛成简单的状态机，方便接入 Kafka、RocketMQ 或自研队列

## 示例说明

示例中的 `Controller` 根据当前快照决定消费者是否应当暂停拉取新消息。测试覆盖了 lag 过高触发暂停、处理中的消息过多触发暂停，以及在指标回落到恢复阈值前保持暂停状态这三个关键行为。
