# 用 deadline clamp 保护下游调用预算

上游请求常常会带着一个总超时进入服务端，但直接把这个 deadline 透传给数据库、缓存或 RPC 客户端，容易出现两个问题：一是剩余预算过小却还继续发请求，白白放大失败流量；二是没有上限，导致慢下游长期占用连接与 goroutine。

## 设计要点

- 为每次下游调用设置明确的 `minBudget` 与 `maxBudget`
- 如果上游剩余时间已经低于最小预算，就直接快速失败
- 即使上游没有 deadline，也主动给下游设置可控 timeout

## 示例说明

示例中的 `Clamp` 会根据父 `context` 的剩余时间和本地策略计算实际预算，并返回一个新的带超时上下文。测试覆盖了无上游 deadline 时使用本地上限、上游预算更小时自动收缩，以及剩余预算不足时直接拒绝发起下游调用。

## 运行方式

```bash
go test ./daily/2026/07/07/deadline_clamp
```
