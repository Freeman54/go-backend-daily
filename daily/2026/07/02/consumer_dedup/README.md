# 用去重表实现消息消费幂等

消息队列至少一次投递时，消费端一定会遇到重复消息。如果业务副作用不可重放，比如发券、扣库存、记账，就需要先做去重，再执行业务处理。

## 设计要点

- 以业务唯一键或消息 ID 作为去重键
- 先写去重记录，再执行有副作用的处理逻辑
- 去重记录带 TTL，兼顾重试窗口和存储成本

## 示例说明

线上实现通常会把 `DedupStore` 接到 Redis、数据库唯一索引或带过期能力的 KV。关键点不是“记过这个消息”，而是“能否原子地判重并占位”。

## 运行方式

```bash
go test ./daily/2026/07/02/consumer_dedup
```
