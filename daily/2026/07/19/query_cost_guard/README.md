# 用 query cost guard 保护数据库免受重查询挤压

数据库瓶颈不只来自“请求数太多”，还来自“少量特别重的查询”。如果系统只按请求个数限流，多个大查询同时进入，依然会把连接池、锁等待和 IO 压力推高。

`query cost guard` 用“成本”而不是“次数”做准入控制。每条查询申请固定成本预算，执行完成后归还预算。这样轻查询可以并行，大查询则会被主动排队。

## 设计要点

- 预算总量固定，重查询会占用更多配额
- `Acquire` 支持 `context`，调用方可以设置等待上限
- `Release` 归还成本预算，让后续查询继续进入

## 示例说明

示例实现提供：

- `New(totalBudget)`：创建总成本预算
- `Acquire(ctx, cost)`：申请成本额度并返回释放函数
- 超过总预算的单条查询会被直接拒绝

适合数据库热点保护、复杂报表接口、后台批任务错峰等场景。

## 运行方式

```bash
go test ./daily/2026/07/19/query_cost_guard
```
