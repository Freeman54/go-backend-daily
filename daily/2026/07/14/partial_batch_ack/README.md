# 批量消费时做部分 ACK，避免整批重试放大失败

消息队列批量拉取后，经常只有少数消息因为脏数据或暂时性错误失败。如果实现上只能“整批成功”或“整批失败”，就会把已经处理成功的消息一遍遍重试，既浪费吞吐，也容易制造重复副作用。

## 设计要点

- `BuildPlan` 按单条消息的处理结果拆成 `ack`、`retry`、`drop` 三类
- `ack` 适合已成功入库的消息，`retry` 留给临时故障，`drop` 留给不可恢复脏数据
- `RetryRatio` 可以直接挂到指标里，帮助判断当前消费者是不是在被异常消息拖慢

## 示例说明

示例把一个批次拆成成功确认、需要重试和直接丢弃三部分。真实系统里可以据此分别调用 MQ SDK 的确认、重投或死信逻辑。
