# 按步骤拆分请求 deadline，避免某个下游吃光全部超时预算

一个 API 往往会顺序访问缓存、数据库、降级服务。如果直接把同一个 `context.Context` 传到底，前面的慢操作很容易把全部 deadline 用完，后面的补救逻辑连尝试机会都没有。更稳妥的做法是先把剩余时间拆成多个步骤预算。

## 设计要点

- `Partition` 从父 `context` 的剩余时间中扣掉预留缓冲，再按权重分配给各步骤
- `reserve` 用来给编码、序列化、回包等尾部动作留一点时间
- 权重越高的步骤拿到越多预算，适合把主要时间留给数据库或核心 RPC

## 示例说明

示例把 1 秒 deadline 拆成 cache、db、fallback 三段，并预留 100ms 作为尾部缓冲。测试验证数据库步骤拿到的预算更大，也验证没有 deadline 时应该立刻报错。

## 运行方式

```bash
go test ./daily/2026/07/14/deadline_partition
```
