# 用 retry classifier 避免把不可恢复错误也反复重试

“发生错误就重试”听上去安全，实际上经常会把永久失败放大成请求风暴。参数校验失败、调用方主动取消、明显不可恢复的业务冲突，继续重试只会徒增负载；而 429、5xx 或暂时性超时，通常又值得保留重试机会。把错误先分类，再决定是否重试，是更稳妥的后端策略。

## 设计要点

- 显式区分永久错误、临时错误和调用方取消
- 支持识别 HTTP 状态、`context` 错误和自定义 `Temporary()` 错误
- 用最大尝试次数封顶，避免长尾失败无限放大

## 示例说明

`Policy.ShouldRetry` 根据错误类型和当前尝试次数做决策。测试覆盖 429/5xx、永久错误、`context.Canceled`、`context.DeadlineExceeded` 和最大重试次数。

## 运行方式
