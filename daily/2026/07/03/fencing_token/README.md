# 用 fencing token 防止过期租约继续写入

分布式锁只解决“谁先拿到锁”，但解决不了暂停中的旧持有者恢复后继续写数据的问题。fencing token 的做法是在每次发放租约时递增版本号，下游写入必须携带最新 token，旧 token 即使晚到也会被拒绝。

## 设计要点

- 每次发放或续租都分配更大的 token，保证全局单调递增
- 当前租约过期后，新的持有者拿到更大的 token，旧持有者写入会被识别为陈旧请求
- 真正执行写入的存储层必须校验 token，而不是只信任加锁节点

## 示例说明

示例实现了一个简单租约管理器，覆盖了并发抢占失败、过期后新租约接管、旧 token 写入被拒绝等场景。它适合类比库存扣减、任务调度器 leader 写状态等路径。

## 运行方式

```bash
go test ./daily/2026/07/03/fencing_token
```
