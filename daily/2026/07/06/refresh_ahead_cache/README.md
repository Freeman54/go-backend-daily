# 用 refresh ahead 在过期前主动预热缓存

缓存命中率高并不代表尾延迟稳定。如果热点对象总是在过期瞬间由请求线程触发回源，就会把一次慢查询直接暴露给用户。更稳妥的策略是在对象临近过期时继续返回旧值，同时异步触发一次预刷新，把真实回源成本挪到用户请求路径之外。

## 设计要点

- 命中热数据时直接返回缓存值，不把预刷新阻塞在用户请求上
- 进入 `refreshAhead` 窗口后只触发一次刷新，避免每个请求都回源
- 真正过期后不再返回值，强制等待刷新结果或走降级路径

## 示例说明

示例实现了一个最小化 `Cache`：`Get` 会根据当前时间判断是直接命中、进入预刷新窗口还是已经过期，并返回是否应该触发刷新。测试覆盖了缓存未命中、健康命中和临近过期只触发一次刷新三种场景。

## 运行方式

```bash
go test ./daily/2026/07/06/refresh_ahead_cache
```
