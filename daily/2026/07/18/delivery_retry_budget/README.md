# 用 delivery retry budget 避免单条坏消息无限消耗重试资源

消息系统里最怕两种情况：一种是临时性错误没有重试，造成可恢复失败被直接丢弃；另一种是结构性坏消息被无限重试，长期占着消费者配额和告警资源。`delivery retry budget` 的做法是为每条消息设置总尝试预算，并结合错误类型决定是继续重试还是直接隔离。

## 设计要点

- 临时错误可以重试，但总次数必须有上限
- 不可恢复错误直接进入隔离区或死信，不参与重试
- 预算耗尽后立刻停止继续投递，避免拖垮整个消费组

## 示例说明

示例中的 `Budget.Decide(attempts, isTransient)` 返回两种结果：

- `Retry`：暂时失败且预算未耗尽
- `Quarantine`：预算耗尽，或错误本身不可恢复

适合 MQ 消费、Webhook 投递、异步任务调度等需要平衡恢复能力与整体吞吐的场景。

## 运行方式

```bash
go test ./daily/2026/07/18/delivery_retry_budget
```
