# 用 write-behind queue 批量刷新缓存而不是同步写入

有些缓存写入动作不一定要和主请求同步完成，例如热点榜单、画像标签、推荐特征或统计快照。每次请求都直接写缓存会放大尾延迟，也可能让外部缓存成为主链路依赖。`write-behind` 的做法是先把更新放入本地或独立队列，再由后台批量 flush。

## 设计要点

- 主请求只负责把变更加入队列，降低同步写延迟
- 后台按批次 flush，减少缓存客户端连接与序列化开销
- 要接受“最终一致”而不是“立即可见”的语义

## 示例说明

示例中的 `Store` 同时维护待刷新的队列和已可见缓存：

- `Enqueue(key, value)` 追加待刷新的更新
- `Flush(limit)` 分批应用更新
- 后写入的同一 key 会覆盖前一个值

适合异步缓存预热、统计类快照、对读时延敏感但允许短暂延迟可见的场景。
