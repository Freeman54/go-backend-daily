# 用乐观锁避免并发写覆盖

订单、配置、库存等对象在后台系统里经常会被多个操作者同时修改。若更新语句不校验版本号，后写入的请求会静默覆盖先前结果。这个示例用内存版 `version` 字段演示乐观锁的核心流程。

## 设计要点

- 读取时拿到当前 `Version`
- 更新时带上 `expectedVersion`，只有版本匹配才允许写入
- 冲突返回显式错误，让上层决定重试、提示用户或回滚流程

## 示例说明

落到数据库里时，常见写法是 `update ... where id = ? and version = ?`。若受影响行数为 0，就说明出现并发写冲突。

## 运行方式

```bash
go test ./daily/2026/07/01/optimistic_lock
```
