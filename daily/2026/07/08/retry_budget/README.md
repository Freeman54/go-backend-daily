# 用 retry budget 限制失败风暴中的重试放大

下游已经在报错时，如果调用方还无上限地继续重试，故障会被放大成更高的流量和更长的恢复时间。`retry budget` 的核心思路是把“允许的重试次数”绑定到最近窗口内的成功量上，只给系统保留有限的补救空间。

## 设计要点

- 每个依赖单独维护窗口，避免一个热点依赖拖垮其他路径
- 基础重试额度保证冷启动或低流量场景仍有少量补救机会
- 成功越多，可用重试越多；成功归零时，重试也会被快速收紧

## 示例说明

示例中的 `Budget` 支持按 key 记录成功次数，并在同一时间窗口内计算可用重试额度。测试覆盖了基础额度、成功量放大预算，以及窗口滚动后额度重置三类后端常见场景。

## 运行方式

```bash
go test ./daily/2026/07/08/retry_budget
```
