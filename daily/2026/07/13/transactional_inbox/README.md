# 用 transactional inbox 控制消息处理租约与重试

消费者把消息拉下来之后，最难处理的是“处理到一半进程挂了”这种情况。没有租约，其他消费者可能永远不敢接手；没有显式重试时间，失败消息又会被立刻反复打爆下游。`transactional inbox` 的核心价值，是把一条消息的处理状态、租约和重试时间放进同一个一致性模型里。

## 设计要点

- `Claim` 为消息设置处理租约，避免多个消费者同时处理同一条记录
- `Fail` 把消息重置为待处理状态，同时写入下次可重试时间
- `Complete` 明确落成终态，后续重复消费直接拒绝

## 示例说明

这个内存版 `Store` 模拟 inbox 表的核心字段和状态机。测试覆盖首次领取、租约未到期时的并发保护，以及失败后延迟重试。

## 运行方式

```bash
go test ./daily/2026/07/13/transactional_inbox
```
