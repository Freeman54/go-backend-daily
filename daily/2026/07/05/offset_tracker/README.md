# 用 offset tracker 安全提交乱序完成的消息消费位点

并发消费一个 partition 时，消息处理完成顺序未必和拉取顺序一致。如果每条消息处理完就直接提交 offset，很容易把中间失败的消息跳过去，后续重启后就永远补不回来了。更安全的策略是只提交“已经连续完成”的最高位点。

## 设计要点

- 允许处理乱序完成，但提交 offset 必须保持连续
- 尚未形成连续区间的 offset 先暂存，等缺口补齐后再一次性推进
- 已经提交过的旧 offset 直接忽略，避免重复回退

## 示例说明

示例实现了一个简单的 `Tracker`，维护下一个待提交 offset 和一组已完成但尚未连续的位点。测试覆盖了乱序确认推进，以及重复确认旧 offset 不回退两种场景。
