# 用副本延迟阈值保护读写一致性

主从或多副本数据库可以分担读流量，但只要复制存在延迟，就会遇到“刚写完马上读却读不到”的一致性问题。实践里通常不会把所有读都打回主库，而是结合读场景和副本延迟做动态路由。

## 设计要点

- 对 read-after-write、结算、库存这类强一致读，直接走主库
- 其余读请求仅在副本健康且延迟低于阈值时才放行
- 多个可用副本中优先选择延迟最小者，避免把请求打到正在抖动的副本

示例实现了一个简单路由器：根据 `readAfterWrite` 标记、副本健康状态和最大允许延迟选择目标节点。测试覆盖了强一致读、副本优选和超阈值回退主库三种场景。
