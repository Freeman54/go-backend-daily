# 后端错误分类与 HTTP 状态码映射

后端服务里最容易失控的一类代码，就是错误处理。刚开始只有 `return err`，后来每层都包一遍，最后 API 网关、日志、监控和前端拿到的是一团看不出边界的错误字符串。

更稳的做法是把错误分成两层：

- 内部错误链：保留真实原因，方便日志、排障和 `errors.Is` / `errors.As` 判断。
- 对外错误语义：暴露稳定的错误码、HTTP 状态码和安全的用户提示。

## 示例目标

这个示例实现了一个很小的错误分类包：

- `Kind` 表示业务语义，例如参数错误、未登录、无权限、资源不存在、冲突、限流和内部错误。
- `AppError` 保存错误分类、对外消息和底层 cause。
- `KindOf` 从任意 wrapped error 中提取错误分类。
- `HTTPStatus` 把错误分类映射到 HTTP 状态码。
- `Public` 生成可以返回给调用方的错误体，并隐藏内部错误细节。

## 关键设计

错误分类应该稳定，不应该直接等于数据库、缓存或 RPC 客户端的错误类型：

```go
type Kind string
```

底层错误仍然通过 `Unwrap` 保留下来：

```go
func (e *AppError) Unwrap() error {
    return e.Cause
}
```

这样业务层可以包装真实原因：

```go
err := Wrap(KindNotFound, "用户不存在", sql.ErrNoRows)
```

HTTP 层只关心错误语义：

```go
status := HTTPStatus(err)
body := Public(err)
```

## 实践建议

1. 错误分类要少而稳定，优先表达调用方能采取什么动作，而不是表达服务内部发生了什么。
2. 日志里记录完整错误链，对外响应只返回错误码和安全消息。
3. 不要把数据库错误、Redis 错误、RPC 错误直接穿透给前端。
4. 内部错误统一映射为 500，并返回泛化提示，避免泄漏表名、字段名、连接串或实现细节。
5. 对于参数错误、资源不存在、状态冲突这类业务错误，可以保留明确的人类可读消息。

## 小结

错误处理不是简单地“把 err 返回上去”。一个后端服务需要同时照顾排障效率、接口稳定性和安全边界。把错误链和对外错误语义拆开，是让服务长期可维护的一步。
