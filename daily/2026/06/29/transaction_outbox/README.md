# 用事务 outbox 保证事件最终投递

“先写数据库，再发消息”最怕数据库成功、消息失败，导致状态和事件不一致。这个示例用内存结构模拟事务 outbox：业务数据和待发送事件一次提交，后续由独立 dispatcher 重试发送。

## 设计要点

- 订单写入和 outbox 事件追加在同一把锁内完成，模拟同事务提交
- `Dispatcher.FlushOnce` 只挑未发布事件，发送成功后再标记已发布
- 发送失败时保留事件，等待下一轮补偿重试

## 示例说明

真实生产环境里，outbox 表通常由定时任务、消息 relay 或 CDC 组件持续扫描。这种模式适合订单、支付、库存等需要最终一致性的场景。

## 运行方式
